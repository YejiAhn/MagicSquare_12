# MagicSquare 4x4 PRD

## 1. 문서 개요

- **문서명**: MagicSquare 4x4 Product Requirements Document
- **프로젝트명**: MagicSquare
- **방법론**: Dual-Track UI + Logic TDD with MLOps
- **문서 목적**: 4x4 매직스퀘어 학습 시스템의 제품 요구사항을 정의하고, 설계/테스트/구현/검증의 기준을 고정한다.

## 2. 문제 정의

### 2.1 표면 문제(잘못된 정의)

- "4x4 마방진 하나를 완성하면 된다."

### 2.2 실제 문제(정확한 정의)

- 4x4 격자 상태가 주어졌을 때, 불변조건(Invariant)과 I/O 계약에 따라 상태의 타당성을 일관되게 판정하고, 두 빈칸의 값을 규칙적으로 결정해 재현 가능한 결과를 반환할 수 있어야 한다.

### 2.3 핵심 가치

- 정답 생성 자체보다 **규칙 기반 사고**, **계약 중심 설계**, **검증 자동화**, **회귀 방지**를 훈련한다.

## 3. 비즈니스 목표 및 성공 지표

### 3.1 Epic

- **Invariant(불변조건) 기반 사고 훈련 시스템 구축**

### 3.2 성공 지표(KPI)

- Domain Logic 테스트 커버리지 95% 이상
- 입력 검증 계약 테스트 100% 통과
- 하드코딩/매직 넘버 금지 규칙 준수
- Invariant -> Rule -> Test 추적 가능성 확보

## 4. 사용자 및 사용 맥락

### 4.1 핵심 사용자(Persona)

- 소프트웨어 개발 학습자
- TDD 훈련 중인 주니어/미드 개발자
- Clean Architecture(ECB) 이해를 높이려는 학습자

### 4.2 사용자 목표

- 구현 이전에 계약과 불변조건을 명확히 정의한다.
- UI/Logic을 분리해 병렬 TDD를 수행한다.
- 리팩터링 이후에도 외부 계약을 깨지 않는다.

## 5. 제품 범위

### 5.1 포함 범위(In Scope)

- 4x4 입력 검증
- 빈칸(0) 2개 좌표 탐색
- 누락 숫자 2개 탐색(1~16 기준)
- 마방진 판정(행/열/대각선 합 = 34)
- 두 조합 시도(정순/역순) 후 결과 반환
- 표준 오류 코드/메시지 반환
- 테스트/추적성 중심 산출물 제공

### 5.2 제외 범위(Out of Scope)

- 일반화된 NxN 마방진 최적화
- 네트워크/권한/인증 기능
- 실서비스용 UI 고도화
- 분산 저장소/트랜잭션 설계

## 6. 핵심 불변조건(Invariants)

- 격자 크기는 항상 4x4
- 값 범위는 0 또는 1~16
- 0 제외 중복 금지
- 0(빈칸)은 정확히 2개
- 완성 상태에서 모든 행/열/두 대각선의 합이 34
- 동일 입력에 대해 동일 출력(결정성)
- 출력 포맷은 항상 `[r1,c1,n1,r2,c2,n2]` (좌표 1-index)

## 7. 계약 요구사항

### 7.1 입력 계약

- 타입: `int[4][4]`
- 제약:
  - 빈칸은 `0`으로 표현
  - 빈칸 개수는 정확히 2개
  - 값은 `0` 또는 `1~16`
  - `0` 제외 중복 금지

### 7.2 출력 계약

- 타입: `int[6]`
- 포맷: `[r1,c1,n1,r2,c2,n2]`
- 좌표는 1-index
- 조합 규칙:
  - `(작은 수 -> 첫 빈칸, 큰 수 -> 둘째 빈칸)`이 성립하면 해당 순서 채택
  - 불성립 시 역순 결과 채택

### 7.3 오류 계약

- 입력 오류: `VALIDATION_ERROR` 계열
- 도메인 성립 실패: `DOMAIN_NO_VALID_PLACEMENT`
- 데이터 계층 오류(선택): `DATA_*`
- 오류 응답은 `code`, `message`, `details`를 포함

## 8. 사용자 여정 및 스토리 요구사항

### 8.1 사용자 여정(Journey)

1. 문제 인식: "구현"이 아닌 "불변조건 기반 검증"으로 목표 재정의
2. 계약 정의: 입력/출력/예외 정책 고정
3. Domain 분리: BlankFinder, MissingNumberFinder, Validator, Solver
4. Dual-Track TDD: UI RED / Logic RED 병렬 -> GREEN 최소 구현 -> REFACTOR
5. 회귀 보호: 엣지/입력오류/조합실패 케이스 확장

### 8.2 필수 사용자 스토리

- Story 1: 입력 검증
  - AC: 4x4 위반, 빈칸 개수 위반, 중복, 범위 위반 시 예외
- Story 2: 빈칸 탐색
  - AC: row-major 순서, 정확히 2개 좌표 반환
- Story 3: 누락 숫자 탐색
  - AC: 1~16 기준 누락 2개, 오름차순 반환
- Story 4: 마방진 판정
  - AC: 모든 행/열/대각선 합 동일(34)
- Story 5: 두 조합 시도
  - AC: 정순 우선, 실패 시 역순, 결과 길이 6 유지

## 9. 기능 요구사항(Functional Requirements)

- FR-01: 시스템은 입력 계약 위반을 도메인 진입 전에 차단해야 한다.
- FR-02: 시스템은 빈칸 좌표 2개를 row-major 순서로 식별해야 한다.
- FR-03: 시스템은 누락 숫자 2개를 계산해 정렬된 형태로 제공해야 한다.
- FR-04: 시스템은 정순/역순 배치를 시도하고 유효한 해를 선택해야 한다.
- FR-05: 시스템은 결과를 1-index 좌표 기반 `int[6]`으로 반환해야 한다.
- FR-06: 시스템은 실패 원인을 표준 오류 코드로 반환해야 한다.

## 10. 비기능 요구사항(Non-Functional Requirements)

- NFR-01: 테스트 우선 개발(TDD) 절차 준수(RED -> GREEN -> REFACTOR)
- NFR-02: ECB 의존성 방향 준수(boundary -> control -> entity)
- NFR-03: 코드 품질 규칙 준수(PEP8, 타입힌트, Google docstring)
- NFR-04: 회귀 테스트 자동화 및 커버리지 목표 달성
- NFR-05: 규칙/계약 변경 시 추적성 매핑 업데이트 필수

## 11. 기술/아키텍처 제약

- 아키텍처 패턴: ECB
- Boundary는 Entity를 직접 호출하지 않는다.
- Domain(Entity/Control)은 상위 계층 구현에 의존하지 않는다.
- Data 계층은 저장/로드 인터페이스 수준에서 교체 가능해야 한다.

## 12. 검증 시나리오(Technical Acceptance)

### 12.1 정상 시나리오

- 작은 수 -> 첫 빈칸 배치로 34 마방진이 완성되면 해당 결과 반환
- 정순 실패/역순 성공 시 역순 결과 반환

### 12.2 실패 시나리오

- 빈칸 개수 오류
- 중복 숫자 존재
- 값 범위 초과
- 두 조합 모두 실패

## 13. 테스트 및 품질 기준

- Domain: 95%+
- UI Boundary: 85%+
- Data: 80%+
- 계약 테스트는 입력/출력/오류 스키마를 고정 보호해야 한다.
- 모든 핵심 Invariant는 최소 1개 이상의 테스트로 보호되어야 한다.

## 14. 추적성(Traceability) 요구

- 추적 경로: `Invariant -> Rule -> Use Case -> Contract -> Test -> Component`
- 릴리스/리뷰 시 위 경로가 깨진 항목이 없어야 한다.
- 신규 규칙 추가 시 대응 테스트/계약 업데이트가 동반되어야 한다.

## 15. 릴리스 완료 기준(Definition of Done)

- 모든 필수 스토리 AC 충족
- Technical 시나리오 통과
- 커버리지 목표 달성
- forbidden 규칙 위반 없음
- 계약 변경 없는 리팩터링 보장

## 16. 리스크 및 대응

- 리스크: "정답 출력" 중심으로 범위가 축소될 위험
  - 대응: 불변조건/계약 테스트를 먼저 고정
- 리스크: 레이어 경계 붕괴 위험
  - 대응: ECB 의존성 검사와 리뷰 체크리스트 운영
- 리스크: 리팩터링 중 계약 훼손
  - 대응: 회귀 테스트 + 계약 테스트 게이트 적용
